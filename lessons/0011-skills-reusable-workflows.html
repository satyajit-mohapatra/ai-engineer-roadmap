Skills

Write a triggerable workflow once. Invoke it in any session, forever.

What a Skill Is

A skill is a Markdown file that Claude reads and executes when you invoke it. You type /skill-name and Claude loads the skill's instructions and runs the workflow described there.

Skills are layer 4 of the harness — they sit above hooks (layer 5) and below the tool surface (layer 2). They're the reusable workflow layer. Anything you find yourself explaining to Claude repeatedly is a skill candidate.

Why Skills Beat Prompts
A prompt lives in your head or clipboard. A skill lives in the harness. It's consistent across sessions, gets refined over time, and works the same whether you trigger it or Claude does. Skills are the Write strategy applied to procedures, not just data.

Anatomy of a Skill File

Skills are stored in ~/.claude/skills/ (global) or .claude/skills/ (project). The file format is Markdown with YAML frontmatter:

~/.claude/skills/my-skill.md---
name: my-skill
description: One-line description — used to decide when to suggest this skill
argument-hint: Optional hint shown in the /skill autocomplete UI
---

# The body is plain Markdown instructions.
# Claude reads this and follows the instructions when the skill is invoked.
# Write it like a clear procedure, not a prompt.

The user has invoked this skill. Follow these steps:

1. Do X
2. Then do Y
3. Output the result as Z

Key frontmatter fields:

FieldPurposeRequired
nameThe trigger slug — what you type after /Yes
descriptionOne-line summary used by Claude to decide when to auto-suggest the skillYes
argument-hintShown in autocomplete UI when the user starts typing /skill-nameNo
disable-model-invocationSet true to prevent the skill from being auto-suggested by ClaudeNo

Where Skills Live

~/.claude/skills/ # global — available in all projects └── my-global-skill.md /your-project/.claude/skills/ # project — only available here └── project-specific.md

Global skills are always available. Project skills are only loaded when Claude Code is running inside that project directory. You already have skills at ~/.claude/skills/teach/ — the /teach skill you triggered to start this lesson.

A Real Skill for Your Project

Here's a generic skill that runs a structured code review whenever you invoke /code-review. Adapt the criteria to your language or framework:

.claude/skills/code-review.md---
name: code-review
description: Review a source file for correctness, performance, and project conventions
argument-hint: Which file should I review? (leave blank to use the most recently edited)
---

The user wants a code review. Follow this procedure:

## Step 1: Identify the target
If the user specified a file, read it. Otherwise, identify the most recently
edited source file in the project src/ directory.

## Step 2: Review against these criteria
Check each of the following and note any issues:
- Naming conventions: match the project's documented style (check CLAUDE.md)
- No unused imports, dead code, or debug statements
- Error handling: all async calls have try/catch or equivalent
- Security: no secrets, no SQL string interpolation, no eval()
- Performance: flag O(n²) loops where a map/set lookup would suffice
- Test coverage: flag any exported function with no corresponding test

## Step 3: Output format
Produce a structured review with:
- **Summary**: 1-2 sentences on overall quality
- **Issues**: bulleted list, severity tagged [HIGH/MED/LOW]
- **Suggestions**: specific improvements with example code
What Makes a Good Skill Body
The best skill bodies are procedures, not prompts. They describe exact steps, output formats, and decision rules. Claude follows them literally — so be specific. Vague instructions in skills produce vague output, just like vague prompts do.

Skills vs. Hooks vs. CLAUDE.md

LayerWhen it runsYou control it by
CLAUDE.mdEvery session, always loadedWriting project rules (layer 1)
SkillsOn demand — when invoked with /skill-nameWriting a skill file (layer 4)
HooksAutomatically at lifecycle eventsConfiguring settings.json (layer 5)

The rule of thumb: if it should run every time, put it in CLAUDE.md. If it should run when triggered, make it a skill. If it should run automatically on an event, make it a hook.

Hands-On: Write Your First Skill

Create the code-review skill in your project:

  1. Create .claude/skills/ directory: mkdir -p /path/to/your-project/.claude/skills
  2. Create the file at .claude/skills/code-review.md with the content above
  3. Adjust the Step 2 criteria to match your project's language and conventions
  4. In a new Claude Code session in this project, type /code-review

A SessionStart hook fires automatically every time a session opens. That's automatic, not on-demand. A skill would work but requires you to remember to type /check-issues — defeating "automatic." CLAUDE.md would tell Claude to check issues, but it won't reliably trigger action unless a hook enforces it.
The key word is "automatically." Automatic = hook. On-demand = skill. Always-loaded context = CLAUDE.md. A SessionStart hook fires every time you open Claude Code — no manual invocation needed.

Correct. The description is read by Claude to decide whether to proactively suggest the skill. A poor description means Claude may never surface the skill at the right moment. Think of it as a semantic trigger: "if the user's task matches this description, suggest this skill."
The description is used by Claude (not just humans) to decide when to auto-suggest the skill. It's a semantic signal — Claude matches the user's task against skill descriptions and surfaces relevant ones. Write it as a clear statement of when this skill applies.

Project-local skills override global ones of the same name. This lets you have a generic global skill and a project-specific override — the same pattern as project settings.json overriding global settings.json. Use this to customize global workflow skeletons for specific projects.
Project-local skills take precedence. This mirrors the settings.json precedence: project overrides global. It lets you maintain a global skill skeleton and override it per-project without touching the global version.

What to Hold Onto

  1. Skills = reusable procedures in Markdown. Write them like step-by-step instructions, not prompts. Specificity is what makes them work reliably.
  2. Trigger: /skill-name. Location: ~/.claude/skills/ (global) or .claude/skills/ (project).
  3. The description field is read by Claude, not just humans. It's the signal Claude uses to auto-suggest the skill. Make it semantic and precise.
  4. Skills vs. Hooks vs. CLAUDE.md: on-demand → skill, automatic → hook, always-loaded context → CLAUDE.md.
