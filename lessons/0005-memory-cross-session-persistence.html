Memory

What Claude knows across sessions — and how you control it.

The Problem Memory Solves

Claude's context window is a session artifact. When you close a session, everything in it disappears. If you had to re-explain your preferences, your project state, and your constraints at the start of every session, the agent would be useless for sustained work.

Memory files solve this. They're plain Markdown files written to disk — a durable Write operation — that Claude reads and injects into future sessions via the Select strategy: only the relevant files are loaded, not all of them at once.

Memory vs. CLAUDE.md
CLAUDE.md: always loaded, stable rules, low-frequency change.
Memory files: selectively loaded, current state, can be updated or deleted. Use memory for anything that changes — project state, your preferences, recent decisions.

Where Memory Lives

For this project, memory files are at:

path~/.claude/projects/-path-to-your-project/memory/
  ├── MEMORY.md          # index — one line per memory file
  ├── user_role.md
  ├── feedback_testing.md
  └── project_migration.md

The MEMORY.md index is always loaded. The individual files are loaded on demand — Claude reads the index to find relevant files, then reads those files when they're relevant to the current task.

Four Memory Types

user
Who you are: role, expertise level, preferences. Helps Claude tailor explanations and recommendations to you specifically.
feedback
How you want to collaborate: corrections ("don't mock the DB"), confirmations ("yes, one bundled PR"). The most impactful type — prevents repeating the same mistakes.
project
Current project state: ongoing migrations, decisions made, deadlines. Decays fast — always include a Why and a date so future sessions can tell if it's still relevant.
reference
Pointers to external resources: Linear project names, Grafana dashboards, Slack channels. Where to look, not what's there.

Memory File Format

memory/feedback_no_trailing_summaries.md---
name: no-trailing-summaries
description: User wants terse responses — no summary paragraph at the end of every reply
metadata:
  type: feedback
---

Don't summarize what you just did at the end of responses.

**Why:** The user said "I can read the diff" — they find trailing summaries
redundant and mildly irritating.

**How to apply:** End responses with the last piece of relevant content.
No "In summary..." or "To recap..." closings.

The Why and How to apply lines are critical for feedback and project memories. They let Claude judge edge cases rather than following the rule blindly.

What NOT to Save

Memory is expensive (loaded into context) and becomes stale. Don't save:

  • Code patterns or architecture — derivable from reading the files
  • Git history or who changed what — git log is authoritative
  • Debugging solutions — the fix is in the code; the commit message has the context
  • Anything already in CLAUDE.md
  • Ephemeral task state (use the conversation context instead)
Stale Memory is Actively Harmful
A memory that says "migration is in progress" six months after it completed will mislead Claude. Treat memory as a living system: update or delete files when state changes. A good memory has a date and a Why — so future Claude can judge relevance.

The MEMORY.md Index

Claude always loads MEMORY.md first. It's a table of contents — one line per memory, under 150 characters each:

MEMORY.md# Memory Index

- [No trailing summaries](feedback_no_trailing_summaries.md) — user finds summary paragraphs redundant
- [User background](user_background.md) — intermediate prompter, new to harness engineering, current project domain
- [ESM migration](project_esm_migration.md) — migrating src/ from CommonJS to ESM, started 2026-06-08

Keep the index under 200 lines — anything after that gets truncated when loaded into context.

What to Hold Onto

  1. Memory = Write + Select. Files are written to disk. MEMORY.md index tells Claude they exist. Individual files are loaded when relevant.
  2. Four types: user, feedback, project, reference. Feedback is the highest-leverage — it's the ratchet.
  3. Always include Why and How to apply in feedback/project memories. Without why, Claude can't judge edge cases.
  4. Memory decays. Prune it. Stale memories are actively harmful — they mislead. Delete or update when state changes.
