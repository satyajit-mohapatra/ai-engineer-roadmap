Context Window Anatomy

What's actually inside the token budget — and how that shapes every engineering decision you've made so far.

Everything Claude Sees Is in the Window

The context window is a finite buffer — typically 200,000 tokens for Claude Sonnet 4.6. Everything that influences Claude's output must be inside it. Nothing outside it has any effect whatsoever. This single constraint explains why every lesson so far has been about what to put in the window and when.

Tool Results: The Hidden Cost

The biggest context consumer in most Claude Code sessions isn't the conversation — it's tool results. A single Bash(cat largefile.js) call can dump 10,000 tokens into the window. A web fetch of a documentation page: 5,000 tokens. A recursive file listing: 2,000 tokens.

The Tool Result Trap
Claude reads the full output of every tool call. If you ask Claude to search 50 files for a pattern and it cats each one, you've burned tens of thousands of tokens on file content. The window fills, quality degrades, and eventually context compaction fires — losing conversation history. Design tool use carefully.

The engineering response to this is the Compress strategy: tools that return summaries instead of raw output, hooks that trim verbose responses, subagents that do targeted lookups instead of broad searches.

The Attention Problem: Lost in the Middle

Research shows that LLM attention isn't uniform across the context window. Information at the very beginning and very end of the window receives the most attention. Content in the middle — especially long middle sections — receives less weight.

Position in contextAttention weightImplication
Start (system prompt, CLAUDE.md)HighPut your most critical rules here
Middle (old conversation, buried tool results)LowerRules buried here may be "forgotten"
End (most recent message)HighRestate critical constraints near your request
Why Short CLAUDE.md Beats Long CLAUDE.md
A 200-token CLAUDE.md where every rule is at the start of the window and salient beats a 2000-token one where important rules get buried. The rule you wrote at line 80 of a long CLAUDE.md has half the effect of the same rule at line 5.

Context Compaction

When the window approaches its limit, Claude Code fires the PreCompact / PostCompact hooks and runs a summarization pass. The oldest conversation history is compressed into a summary, freeing space for new content.

This is automatic, but you can engineer around it:

  • Write important decisions to disk before they get compacted — the Write strategy
  • Keep tool results small — the Compress strategy applied to outputs
  • Use subagents for large searches — they have their own isolated windows (Isolate)

A targeted subagent uses Compress + Isolate. It greps for relevant symbols (compress: don't read what you don't need), works in its own isolated context (isolate: doesn't bloat your main session), and returns only a summary. Reading all files first dumps the entire codebase into your main session's window — often 50,000+ tokens that stay there for the rest of the session.
Spawning a targeted subagent is the right call. It applies Compress (grep first, read only relevant files) and Isolate (its own context window, doesn't bloat your main session). Reading all files in the main session can dump 50,000+ tokens into the window and push earlier content into the "lost in the middle" zone.

Compaction produces a summary — compressed, lossy. The original messages are gone. Details that weren't captured in the summary are permanently lost from the window. This is why the Write strategy (saving decisions to disk before they might be compacted) is critical for long sessions. The harness doesn't automatically save anything — that's your job as the engineer.
Compaction summarizes — lossy compression. Original detail is gone from the window. Critical decisions or state that wasn't written to disk beforehand can be lost. This is the Write strategy's most important use case: checkpoint important state to disk so compaction can't erase it.

What to Hold Onto

  1. The window is everything. Nothing outside it affects output. Every engineering decision is about what earns a place in the window.
  2. Tool results are the biggest consumer. Read ranges, grep before reading, summarize verbose output. Design tools to return compressed, relevant output.
  3. Lost in the middle is real. Start and end of context get most attention. Keep CLAUDE.md short and restate critical constraints near your requests.
  4. Compaction is lossy. Write important state to disk before long sessions. The Write strategy exists because the window doesn't.
