Model vs. Harness

The mental model that separates LLM users from LLM engineers.

The Equation

Every AI agent — including every Claude Code session you've ever run — can be decomposed into exactly two things:

HARNESS
+
MODEL
=
AGENT

Agent = Model + Harness  ·  Addy Osmani, 2026

The model is the neural network. You can't change it (unless you fine-tune, which is out of scope here). What you can change — entirely — is the harness.

Key Insight
A decent model with a great harness beats a great model with a bad harness. The bottleneck in most real Claude Code setups isn't the model — it's the harness.

What Is the Harness?

The harness is every layer of code, configuration, and convention that wraps around the model. Six layers, stacked:

1. System prompt / CLAUDE.md — what Claude knows about your project
2. Tool surface — what actions Claude can take (bash, read, write…)
3. MCP servers — external systems wired in (GitHub, Jira, APIs)
4. Skills — reusable, triggerable workflows
5. Hooks — lifecycle enforcement (pre/post)
6. Memory — what persists across sessions
MODEL

Each layer shapes what the model sees, what it can do, and what it remembers.

Context engineering lives mostly in layers 1, 3, and 6 — what information reaches the model. Harness engineering is all six layers together.

Context Engineering vs. Prompt Engineering

You already write prompts. Here's how context engineering is different:

Dimension Prompt Engineering Context Engineering
Scope One request, one query All inputs: system msg, retrieved docs, tool outputs, memory, conversation history
Timing Written once, reused as-is Assembled dynamically at runtime per request
Question it answers "How should I phrase this?" "What does Claude need to know, right now, to do this?"
Tools Text only RAG, memory, tool descriptions, conversation compaction
Insight
Context engineering is upstream of prompting. A well-engineered context makes the prompt almost irrelevant — the model has what it needs before you even ask.

The Four Context Strategies

There are four operations you can perform on context. Every context engineering decision maps to one of these four strategies:

Write Select Compress Isolate

Write

Persist information outside the context window so it can be retrieved later. In Claude Code, this is your memory files (~/.claude/projects/.../memory/), CLAUDE.md, and any files Claude writes to disk.

Select

Choose which information to inject at inference time. Claude Code skills do this — they only load when triggered, rather than bloating every prompt. RAG systems do this with vector search.

Compress

Reduce token count without losing meaning. Claude Code does this automatically (context compaction). You can engineer this explicitly by writing good summaries in memory files instead of raw transcripts.

Isolate

Run sub-tasks in their own context so they don't contaminate each other. Subagents in Claude Code are isolation — each spawned Agent() call gets a fresh context. This is how long-horizon work stays coherent.

Applied to Your Project
Your CLAUDE.md is a Write operation — it injects project knowledge into every session. Your memory files at ~/.claude/projects/.../memory/ are also Write. If you spawned a subagent to do research, that's Isolate.

What to Hold Onto

Three things from this lesson to keep in mind going forward:

  1. Agent = Model + Harness. You control the harness entirely. The model is fixed. All engineering leverage lives on the harness side.
  2. Context engineering is about the right information at the right time. Not phrasing — information. What does Claude need to know, right now, to do this well?
  3. Four operations: Write, Select, Compress, Isolate. Every context decision is one of these. When something goes wrong, ask: which operation is missing?
