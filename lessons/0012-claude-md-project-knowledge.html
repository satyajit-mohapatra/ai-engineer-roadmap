Lesson 12 Harness Layer 1 ~15 min

CLAUDE.md

The always-loaded project knowledge layer. What to put in, what to leave out, and how to write it well.

What CLAUDE.md Is

CLAUDE.md is a Markdown file that Claude Code reads at the start of every session. It's layer 1 of the harness — the foundation of project-specific knowledge. Everything in it lands in Claude's context window before you type your first message.

It's the Write strategy applied to project rules: you write the knowledge once, the harness injects it every session.

Key Distinction
CLAUDE.md is for knowledge Claude can't derive from reading the code — your WHY, your constraints, your conventions, your workflow commands. Don't use it to describe what's already obvious from the files.

Where Claude Looks

Claude Code loads CLAUDE.md files from multiple locations, stacking them:

LocationScopeUse for
~/.claude/CLAUDE.mdGlobal — every projectYour personal preferences, always-true rules, identity
/project/CLAUDE.mdProject rootProject structure, commands, conventions, team rules
/project/src/CLAUDE.mdSubdirectoryRules specific to one subsystem (e.g. "this dir is read-only")

They're all loaded and concatenated. More specific files (deeper in the directory tree) can override or extend the root file.

What Belongs in CLAUDE.md

Your existing CLAUDE.md is a strong example. Here's the principle behind each section type:

Include
Run commands (npm start, npm test) Project structure that's non-obvious from files Naming conventions (camelCase, UPPER_CASE) Security rules ("never commit .env") Architectural constraints ("use CommonJS, not ESM") Workflow steps (branch → test → lint → push) External docs links (framework docs, test runner docs) What NOT to do (anti-patterns specific to this codebase)
Exclude
Things derivable from reading the code Git history / recent changes In-progress task state Long lists of every file and what it does Generic best practices Claude already knows Things that change frequently (put those in memory)

Token Cost Is Real

Every byte of CLAUDE.md costs tokens on every message. A 500-token CLAUDE.md injected into a 50-session project costs 25,000 tokens just in overhead — before any actual work. This is why you should:

  • Keep CLAUDE.md focused and scannable
  • Move project-state information to memory files (they're selected, not always-loaded)
  • Use subdirectory CLAUDE.md files to scope rules that only apply in one area
Insight: CLAUDE.md is a Write + Select Tradeoff
CLAUDE.md is a pure Write operation — it's always injected. Memory files are Write + Select — written once, loaded only when relevant. For stable, always-needed rules: CLAUDE.md. For project state that changes: memory.

A Well-Structured CLAUDE.md Section

Compare these two ways of describing the same thing:

WEAK — too vague## Code Style
Use good naming conventions and follow best practices.
Make sure tests pass before committing.

STRONG — specific and actionable## Code Style
- Files: camelCase.js  |  Functions: camelCase()  |  Constants: UPPER_CASE  |  Classes: PascalCase
- No comments unless the WHY is non-obvious — never describe WHAT the code does
- No trailing console.log() or debug statements in committed code

## Before Every Commit
Run in order: npm run lint && npm run test
Both must pass. If either fails, fix before committing — never use --no-verify.

The strong version gives Claude exact rules to follow. Claude can check against them. The weak version gives Claude room to guess — and guessing leads to inconsistency.

A
CLAUDE.md — update the module format rule immediately
B
Memory file (project type) — it's an ongoing project state, not a stable rule
C

D
A comment in package.json
A migration is project state — it has a beginning, middle, and end. Memory files are right for this: they capture current state, can be updated as the migration progresses, and won't bloat CLAUDE.md with something temporary. Once migration is done, update CLAUDE.md to say "use ESM" and delete the memory entry.
Migrations are temporary project state — they belong in a memory file, not CLAUDE.md. CLAUDE.md is for stable, long-term rules. Memory files track current state and can be updated or deleted when things change.

Quiz · Question 3 of 3

A
CLAUDE.md was not loaded — the rule wasn't injected

Context window attention degrades for information buried in long documents — the "lost in the middle" problem. If CLAUDE.md is bloated with non-essential content, important rules lose weight. The fix: keep CLAUDE.md focused and short so every rule stays salient. Long CLAUDE.md files are often worse than short ones.
This is likely the "lost in the middle" problem — when CLAUDE.md gets long, rules buried in the middle lose attention weight. The model loaded it, but the rule didn't stay salient. Keep CLAUDE.md short and focused so every rule carries weight.

What to Hold Onto

  1. CLAUDE.md = always-loaded Write operation. Every token costs. Keep it focused on what's non-derivable and stable.
  2. Three levels: global → project root → subdirectory. Use subdirectory CLAUDE.md files to scope rules to where they matter.
  3. Specific beats vague. Give Claude exact rules it can check against, not principles it has to interpret.
  4. Short beats long. A 200-token CLAUDE.md where every rule is salient beats a 2000-token one where important rules get lost.
