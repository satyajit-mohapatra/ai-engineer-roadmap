Vector Databases

Where embeddings live and get searched at scale. The storage and retrieval engine behind every RAG system.

The problem they solve
You have a million document embeddings (Lesson 22). A query comes in. Comparing it to all million with cosine similarity is too slow. A vector database indexes the vectors so it can find the nearest neighbors in milliseconds — approximately, but fast.

The key trick: approximate nearest neighbor (ANN)

Exact nearest-neighbor search over millions of high-dimensional vectors is expensive. Vector DBs use ANN algorithms (most commonly HNSW — a navigable graph) to find almost the closest vectors far faster, trading a sliver of recall for orders-of-magnitude speed. You tune the recall/speed knob per use case.

Indexing vs querying
Indexing = inserting your embeddings and building the ANN structure (done once, or as data arrives). Querying = embedding the incoming query and asking for its k nearest neighbors (every request). Most DBs also store metadata alongside each vector so you can filter (e.g. "only chunks from 2026 reports").

Don't over-engineer the choice
For learning and small projects, Chroma or LanceDB locally is plenty — no account, no server. Reach for Pinecone/Qdrant when you have real scale, multi-user concurrency, or ops requirements. The retrieval logic is nearly identical across all of them; you're choosing operations, not capability.

The shape of using one

python — Chroma, the whole loopimport chromadb
db = chromadb.Client()
col = db.create_collection("reports")

# INDEX: store text + embeddings + metadata
col.add(documents=chunks, embeddings=vecs,
        metadatas=metas, ids=ids)

# QUERY: find the 4 nearest chunks to a question
hits = col.query(query_embeddings=[embed(question)],
                 n_results=4,
                 where={"year": 2026})   # metadata filter

Note the same three concepts in every vector DB: add (index), query (k-NN search), and metadata filtering. Learn them once here; Lesson 24 wires this into a full RAG pipeline.

Check Your Understanding

Quiz · Question 1 of 2
Why do vector databases use approximate nearest neighbor (ANN) search like HNSW instead of just computing exact cosine similarity against every stored vector?
A
ANN is more accurate than exact search because it considers more vectors at once
B
Exact search over millions of vectors is too slow; ANN trades a little recall for huge speed
C
ANN avoids needing embeddings entirely, so it works on raw text directly
D
Exact cosine similarity only works for vectors under 100 dimensions
Correct. ANN exists for speed at scale: comparing a query to millions of vectors exactly is too slow, so algorithms like HNSW find almost the nearest neighbors much faster, sacrificing a small amount of recall. ANN is not more accurate than exact search (it's slightly less), still needs embeddings, and exact cosine works at any dimension — just slowly.
ANN is a speed/recall trade (option B): exact search over millions of vectors is too slow, so HNSW gets almost the nearest neighbors fast, giving up a touch of recall. ANN is slightly less accurate than exact, still requires embeddings, and exact search isn't dimension-capped.
Quiz · Question 2 of 2
You're learning RAG with ~2,000 chunks of your own trading docs on your laptop, no other users. Which vector store fits best, and why?
A
A managed Pinecone cluster, because production-grade infra is always the safer default
B
Chroma or LanceDB locally — no server or account, identical retrieval logic, zero ops
C
A self-hosted Milvus cluster on Kubernetes, to learn the hardest setup first
D
No vector DB at all; 2,000 chunks must be re-embedded on every single query
Correct. At a couple thousand chunks, single-user, on a laptop, an embedded store like Chroma or LanceDB is ideal — no account, no server, and the add/query/filter logic is the same one you'd use in production. Managed clusters and Kubernetes are operational overhead you don't need yet; and you never re-embed stored documents per query (you embed once and store).
For local, single-user, small-scale learning, Chroma or LanceDB (option B) is the right fit — zero ops, same retrieval API as the big tools. Pinecone/Milvus are operational overkill here, and you index documents once rather than re-embedding them on every query.
