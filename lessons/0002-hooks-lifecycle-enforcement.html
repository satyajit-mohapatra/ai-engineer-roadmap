Hooks

Run shell commands at lifecycle events. Block actions before they happen. Enforce quality automatically.

What Hooks Are

A hook is a shell command configured in settings.json that Claude Code fires at a specific lifecycle event. Every time Claude takes an action — edits a file, runs a command, finishes a task — the harness emits events. Hooks intercept them.

Key Insight
You already have hooks. Your global ~/.claude/settings.json has hooks on every event — they're all async monitoring hooks that post JSON to the claude-view live monitor. That's a harness in action. This lesson teaches you to write hooks that enforce, not just observe.

There are two fundamentally different kinds of hook:

KindHow it worksUse for
Blocking Exit code 1 cancels the action before it runs. Claude sees your stdout as an error message. Safety gates: block dangerous git commands, prevent committing secrets, enforce naming rules
Side-effect Runs after the action. Exit code ignored. Can be async. Automation: run linting after JS edits, notify Slack on session end, log all tool calls

The Event Lifecycle

Every Claude Code session fires these events in order. The events marked can block support blocking; the rest are side-effect only.

EventWhen it firesType
SessionStartSession opensinfo
UserPromptSubmitUser sends a messagecan block
PreToolUseBefore any tool call (Bash, Edit, Write, Agent…)can block
PostToolUseAfter any tool call completesside-effect
PostToolUseFailureAfter a tool call errorsside-effect
NotificationClaude sends a notification to the userside-effect
StopClaude finishes a turnside-effect
PreCompactBefore context is compactedside-effect
PostCompactAfter context compactionside-effect
SessionEndSession closesside-effect
SubagentStart / SubagentStopChild agent spawns or endsinfo

The two events you'll use 90% of the time are PreToolUse (for blocking) and PostToolUse (for automation).

The stdin/stdout/exit Protocol

When a hook fires, Claude Code sends a JSON payload to your command's stdin. Your command reads it, decides what to do, and communicates back via stdout and exit code.

What arrives on stdin (PreToolUse example)

{
  "tool_name": "Bash",              ← which tool Claude wants to run
  "tool_input": {
    "command": "git push --force origin main",  ← the exact command string
    "description": "Force push to main"
  }
}

For an Edit tool call, tool_input contains file_path, old_string, new_string. For Write, it contains file_path and content. The structure matches the tool's parameter schema exactly.

How you respond

Exit codeStdoutEffect
0anythingAction proceeds normally
1error messageAction blocked. Claude sees your stdout as an explanation of why.
2anythingHook failed — action proceeds anyway (hook errors don't block)
Why This Design Is Clever
When you exit 1 and print a message, Claude reads that message as context and can try a different approach. You're not just blocking — you're teaching the agent what to do instead. The hook output goes directly into the agent's context window.

The settings.json Schema

Hooks are configured in ~/.claude/settings.json (global, all projects) or .claude/settings.json / .claude/settings.local.json (project-only). The structure:

settings.json{
  "hooks": {
    "PreToolUse": [          // event name
      {
        "matcher": "Bash",      // optional: only fire for this tool
        "hooks": [
          {
            "type": "command",
            "command": "bash /path/to/hook.sh",
            "statusMessage": "Running safety check",  // shown in UI
            "async": false               // false = blocking, true = side-effect
          }
        ]
      }
    ]
  }
}

Key fields:

  • matcher — optional tool name filter ("Bash", "Edit", "Write"…). Omit to match all tools.
  • async: false — Claude waits for the hook. This is how you block.
  • async: true — Claude doesn't wait. Good for logging and notifications (like your claude-view hooks).
  • statusMessage — shown in the Claude Code UI while the hook runs.

Two Real Hooks for This Project

Hook 1 — Block force-push to main (PreToolUse, blocking)

This protects your main branch from being force-pushed by Claude. Create this file:

~/.claude/hooks/guard-git.sh#!/usr/bin/env bash
# Read the PreToolUse JSON from stdin
INPUT=$(cat)

TOOL=$(echo "$INPUT" | jq -r '.tool_name')
CMD=$(echo "$INPUT"  | jq -r '.tool_input.command // empty')

# Only inspect Bash tool calls
[ "$TOOL" != "Bash" ] && exit 0

# Block: git push --force to main/master
if echo "$CMD" | grep -qE 'git push.*(--force|-f).*(main|master)'; then
  echo "BLOCKED: Force-pushing to main is not allowed."
  echo "Use a feature branch and open a PR instead."
  exit 1
fi

exit 0

Then wire it in .claude/settings.json (project level):

.claude/settings.json{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "bash ~/.claude/hooks/guard-git.sh",
          "statusMessage": "Git safety check",
          "async": false
        }]
      }
    ]
  }
}

Hook 2 — Auto-lint JS files after edits (PostToolUse, side-effect)

After Claude edits any .js file in your project, run ESLint automatically (adjust the file extension and linter command for your stack):

~/.claude/hooks/auto-lint.sh#!/usr/bin/env bash
INPUT=$(cat)

TOOL=$(echo "$INPUT" | jq -r '.tool_name')
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

# Only fire for Edit/Write tools on .js files
[[ "$TOOL" != "Edit" && "$TOOL" != "Write" ]] && exit 0
[[ "$FILE" != *.js ]] && exit 0

cd /path/to/your-project
npx eslint --fix "$FILE" 2>&1
exit 0  # PostToolUse exit code doesn't block
.claude/settings.json (add to existing hooks){
  "hooks": {
    "PostToolUse": [
      {
        "hooks": [{
          "type": "command",
          "command": "bash ~/.claude/hooks/auto-lint.sh",
          "statusMessage": "Auto-lint",
          "async": true
        }]
      }
    ]
  }
}

Hands-On: Install Your First Hook

Follow these steps to install the git safety hook. Each step takes under a minute.

1
Create the hooks directory
mkdir -p ~/.claude/hooks
2
Create the guard script
Copy the guard-git.sh script from the "Hook 1" section above into ~/.claude/hooks/guard-git.sh. Make it executable: chmod +x ~/.claude/hooks/guard-git.sh
3
Test it manually
Run: echo '{"tool_name":"Bash","tool_input":{"command":"git push --force origin main"}}' | bash ~/.claude/hooks/guard-git.sh
You should see the BLOCKED message and exit code 1.
4
Add to .claude/settings.json
Create /path/to/your-project/.claude/settings.json with the PreToolUse hook config from the example above. (Your existing settings.local.json has permissions — put hooks in the shared settings.json.)
5
Verify in Claude Code
Start a new session. Ask Claude to run: git push --force origin main. It should be blocked with your error message before Claude can execute it.
Exercise Complete When:
The force-push command is blocked by your hook and Claude receives the "Use a feature branch" message. Ask Claude to try it — then watch the hook intercept the call.

Check Your Understanding

Quiz · Question 1 of 3
You want Claude to automatically run npm test every time it edits a .test.js file. Which hook event and async value should you use?
A
PreToolUse, async: false — run tests before the edit lands
B
PostToolUse, async: true — run tests after the edit, without blocking
C
PostToolUse, async: false — run tests after the edit, blocking Claude until done
D
Stop, async: true — run tests when Claude finishes the whole turn
Correct. PostToolUse fires after the edit is written. async: true lets Claude continue working while tests run in the background — better UX. If you wanted Claude to see the test results before continuing, async: false would make sense, but for automation, async is preferred.
Think about timing and blocking. You want the tests to run after the file is written (not before), and you don't want to block Claude's next action while tests run. PostToolUse + async: true is the right combination.
Quiz · Question 2 of 3
Your PreToolUse hook prints "File is read-only" to stdout and exits with code 1. What happens next?
A
The action proceeds anyway — exit code 1 just means the hook had an error
B
The action is blocked. Claude sees "File is read-only" as context and may try a different approach
C
Claude Code crashes
D
The action is blocked silently — Claude gets no information about why
Exactly. Exit 1 blocks the action AND your stdout message goes into Claude's context window. This is what makes hooks powerful: you're not just saying "no" — you're giving the agent information to work with. A good block message often leads Claude to a correct alternative on its own.
Exit code 1 from a PreToolUse hook blocks the action and feeds your stdout to Claude as an error message. Exit code 2 would be "hook failed, proceed anyway." Exit code 0 is "allowed."
Quiz · Question 3 of 3
You want a hook that fires specifically when Claude uses the Write tool, but not Edit or Bash. What field controls this?
A
The "type" field in the hook object
B
The "matcher" field on the hook entry — set it to "Write"
C
Read tool_name from stdin and filter inside your script
D
Both B and C work — they're equivalent
The matcher field is the right answer — it filters at the harness level before your script even runs. That said, C also works (checking tool_name in your script), and is necessary for more complex conditions. But for a simple per-tool filter, matcher is cleaner and more explicit in the config.
The matcher field filters by tool name before your hook script even runs. You can also filter inside the script by checking tool_name from stdin — both work — but matcher is the declarative, config-level approach. D is partially right but matcher is simpler for this case.

What to Hold Onto

  1. Hooks are the harness's enforcement layer. They run at lifecycle events. PreToolUse can block. PostToolUse runs side effects.
  2. Exit 0 = allow, exit 1 = block + message to Claude, exit 2 = hook error (proceed). Your stdout on exit 1 goes directly into Claude's context.
  3. async: false means Claude waits. async: true means Claude continues. Use false for gates, true for automation.
  4. The ratchet principle in action: every time Claude does something wrong, a hook can permanently prevent it. The harness only gets tighter.
