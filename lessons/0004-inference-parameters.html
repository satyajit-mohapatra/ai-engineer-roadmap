Inference Parameters

The dials on the sampling step. Most people set them by superstition. You're going to set them by knowing what each one does.

Recall from Lesson 2
Generation = repeatedly sampling the next token from a probability distribution. These parameters control how that sample is drawn. They change nothing about the model's knowledge — only its willingness to take the less-likely path.

Temperature — the randomness dial

Temperature reshapes the probability distribution before sampling. Low temperature sharpens it (the top token dominates → deterministic, focused). High temperature flattens it (unlikely tokens get a real chance → creative, varied, riskier).

TemperatureBehaviorUse for
0.0Near-deterministic, always picks the likeliest pathExtraction, classification, code, structured output, math
0.2–0.5Mostly focused, slight variationSummaries, factual Q&A, RAG answers
0.7–1.0Creative, diverseBrainstorming, marketing copy, fiction
>1.0Often incoherentRarely — experiments only
Common myth
Temperature 0 is not guaranteed bit-for-bit deterministic in production — batching, hardware, and mixture-of-experts routing introduce tiny nondeterminism. It's "as deterministic as you can get," not a promise.

Top-p and Top-k — the truncation dials

These limit which tokens are eligible before sampling:

  • Top-k: only consider the k most-likely tokens. top_k=1 = greedy (always the single best).
  • Top-p (nucleus): consider the smallest set of tokens whose cumulative probability ≥ p. top_p=0.9 = "the tokens that make up 90% of the probability mass." Adapts to how confident the model is at each step.
Rule of thumb
Tune temperature OR top-p, not both at once — they interact and make results hard to reason about. Most APIs default to leaving one at its neutral value. Top-p ≈ 0.9–1.0 is a safe default; reach for temperature as your primary creativity dial.

Frequency, presence & repetition penalties

ParameterWhat it discourages
frequency_penaltyReusing the same tokens (scales with how often they've appeared) — reduces verbatim repetition
presence_penaltyReusing any already-seen token at all — pushes toward new topics/vocabulary
repetition_penaltyOpen-source equivalent (e.g. in Ollama/HF) — multiplicatively dampens repeated tokens

Default these to 0. Bump them slightly (0.1–0.5) only if you see the model looping or padding.

Max tokens & stop sequences

  • max_tokens caps the output length (not the prompt). Hit the cap and the response is truncated mid-sentence — set it generously for long answers, tightly for cost control on short ones.
  • stop sequences end generation when a string appears (e.g. "\n\n" or "###"). Useful for fixed formats and to prevent the model from continuing past where you want it.
python — a deliberate config for RAG answersfrom openai import OpenAI
client = OpenAI()
resp = client.chat.completions.create(
    model="gpt-5",
    messages=[...],
    temperature=0.2,        # grounded, low creativity
    max_tokens=500,         # bound the cost
    stop=["\n\nSources:"],   # stop before it invents a sources list
)

Quiz · Question 1 of 2
You're building a function that extracts a stock ticker and a numeric price from messy text and returns strict JSON. What temperature, and why?
A
0.9 — high temperature gives the model freedom to handle messy edge cases better
B
0.0 — extraction wants the single most-likely parse, deterministic and repeatable
C
0.5 — a middle value is always the safest choice for any structured task
D
1.5 — maximum creativity ensures the JSON formatting stays varied and robust
Correct. Extraction and structured output want the single most-likely parse — there's no value in creativity, and randomness only risks malformed JSON or inconsistent fields across runs. Temperature 0 gives you the most focused, repeatable behavior. Creativity (higher temps) is for generation tasks, not parsing.
Extraction → temperature 0 (option B). You want the single most-likely, deterministic parse. Higher temperatures inject randomness that produces inconsistent fields and malformed JSON across runs — the opposite of what a strict extractor needs.
Quiz · Question 2 of 2
A model keeps repeating the same phrase over and over in long outputs. Which adjustment most directly targets that, and what's the standard caution?
A
Raise max_tokens; the repetition is just the model running out of room to vary
B
Add a small frequency penalty; caution — push it too high and output gets incoherent
C
Set top_k to 1; greedy decoding eliminates all repetition by definition
D
Tune temperature and top-p together aggressively for maximum effect
Correct. frequency_penalty directly discourages reusing the same tokens, which is exactly the loop you're seeing. The caution: too high and the model avoids necessary words, producing strained or incoherent text. Greedy decoding (top_k=1) often causes repetition rather than fixing it, and tuning temperature + top-p together is the thing to avoid.
A small frequency_penalty directly discourages reusing the same tokens (option B), with the caution that overdoing it makes output incoherent. More tokens won't help; top_k=1 (greedy) often worsens looping; and tuning temperature + top-p together is explicitly discouraged.
