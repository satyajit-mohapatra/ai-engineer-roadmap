AI Agents & Function Calling

When the model can call your code, it stops being a text generator and becomes a doer. You've used an agent (Claude Code) — now build the mechanism.

Agent, defined
An agent is an LLM in a loop with access to tools, working toward a goal: it reasons, decides which tool to call, reads the result, and repeats until done. That loop is the ReAct pattern from Lesson 7 — Claude Code is one industrial-strength instance of it.

Function (tool) calling — the primitive

You describe functions to the model as JSON Schema. The model doesn't run them — it returns a structured request to call one, with arguments. Your code executes it and feeds the result back. That hand-off is the entire mechanism.

python — defining a tool and handling the calltools = [{
  "type": "function",
  "function": {
    "name": "get_option_price",
    "description": "Get the current price of an option contract",
    "parameters": {"type":"object",
      "properties":{"symbol":{"type":"string"}},
      "required":["symbol"]}}}]

resp = client.chat.completions.create(model="gpt-5",
    messages=msgs, tools=tools)

call = resp.choices[0].message.tool_calls[0]   # model wants a tool
result = get_option_price(**json.loads(call.function.arguments))
msgs.append({"role":"tool", "tool_call_id":call.id,
             "content": str(result)})       # feed result back
final = client.chat.completions.create(model="gpt-5", messages=msgs)

The agent loop

1
Reason
Model decides: answer directly, or call a tool?
↓
2
Act
Model emits a tool call; your code runs it
↓
3
Observe
Tool result is appended to the context
↺
4
Repeat / finish
Loop until the model produces a final answer (or hits a guard rail)
You already know the hard parts
Every reliability concern here — tool surface design, context bloat from tool results, isolation of sub-tasks, knowing when to stop — is what Phase 1 was about. An agent is a harness (Lesson 9). Function calling is just the API-level primitive underneath the harness you've been engineering all along.

Memory & multi-agent systems

  • External memory — agents need state across steps and sessions: scratchpad files, a vector DB of past interactions (RAG over memory), or a structured store. This is the Write/Select strategies (Lesson 13/16) at the agent level.
  • Multi-agent — split a hard job across specialized agents (planner, researcher, coder, critic) that pass messages. Powerful, but adds coordination cost and failure modes — same maker/checker logic as Lesson 21. Start single-agent; reach for multi only when one agent's context can't hold the job.

Quiz · Question 1 of 2
When a model "calls a function," what actually happens at the API level?
A
The model executes your function on the provider's servers and returns its output
B
The model returns a structured request naming the function and arguments; your code runs it
C
The model rewrites your function's source code and ships an optimized version back
D
The function is embedded into the model's weights so future calls are instant
Correct. The model never executes anything — it emits a structured tool call (function name + JSON arguments). Your code runs the function and appends the result to the conversation, then the model continues. This client-side execution is what makes function calling safe and flexible: you control exactly what the tools can do (least privilege, Lesson 8).
The model only requests a call — it returns a structured name + arguments, and your code executes the function and feeds back the result (option B). The provider doesn't run your code, rewrite it, or bake it into weights.
