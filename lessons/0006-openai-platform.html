The OpenAI Platform

Your first direct API. The patterns here — messages, streaming, structured output, tool calls — transfer to every other provider.

Why OpenAI first
Not because it's "best" — because its API became the de-facto shape everyone copies. Learn this once and Anthropic, Gemini, Mistral, and "OpenAI-compatible" endpoints all feel familiar. The roadmap treats it as the reference platform for the same reason.

Chat Completions vs the Responses API

OpenAI has two text APIs. Know which you're looking at:

  • Chat Completions (/v1/chat/completions) — the original, ubiquitous standard. A list of messages with roles (system, user, assistant) in, one message out. This is the format every other provider cloned.
  • Responses API (/v1/responses) — OpenAI's newer, agent-oriented API. Built-in tools, state, and reasoning handling. Preferred for new agentic apps, but Chat Completions remains the lingua franca.
python — the canonical callfrom openai import OpenAI
client = OpenAI()  # reads OPENAI_API_KEY from env

resp = client.chat.completions.create(
    model="gpt-5",
    messages=[
        {"role": "system", "content": "You are a terse options analyst."},
        {"role": "user",   "content": "Explain theta decay in one sentence."},
    ],
)
print(resp.choices[0].message.content)
The three roles, mapped to what you know
system = the persistent instructions (your CLAUDE.md / system prompt from Lesson 12). user = the request. assistant = the model's turns, including prior ones you replay to give it conversation history (since it has no memory — Lesson 2).

Streaming responses

Pass stream=True to get tokens as they're generated instead of waiting for the full response. Essential for chat UX — the user sees output immediately. You iterate over chunks:

pythonstream = client.chat.completions.create(
    model="gpt-5", messages=msgs, stream=True)
for chunk in stream:
    delta = chunk.choices[0].delta.content or ""
    print(delta, end="", flush=True)

Structured output — JSON you can trust

Instead of begging the model for valid JSON and parsing hopefully, pin the output to a schema. With Structured Outputs, the API guarantees the response matches your JSON Schema — no more malformed-JSON retries.

python — guaranteed shapefrom pydantic import BaseModel
class Trade(BaseModel):
    ticker: str
    side: str
    price: float

resp = client.chat.completions.parse(
    model="gpt-5", messages=msgs, response_format=Trade)
trade = resp.choices[0].message.parsed  # a typed Trade object
This is the single biggest reliability upgrade
Structured output turns "LLM in a pipeline" from flaky to dependable. Combine it with temperature 0 (Lesson 4) for extraction tasks and you get near-deterministic, schema-valid data every call.

Function calling & prompt caching (preview)

  • Function / tool calling — you describe functions (name, params as JSON Schema); the model returns a structured request to call one. This is the foundation of agents — full treatment in Lesson 26.
  • Prompt caching — repeated prefixes (a big system prompt, a long document) get cached server-side, cutting cost and latency on follow-up calls. You met the idea in Lesson 19; every major provider now offers it.
  • Vision & images — the same API accepts image inputs (vision) and there are image-generation endpoints (DALL·E) — covered in Lesson 27.

Tokens, pricing & usage

Every response includes a usage object: prompt_tokens, completion_tokens, total_tokens. You're billed separately for input and output (output is usually pricier). Watch this in development — it's how you catch a prompt that quietly ballooned. Add an end-user identifier (user field) to help with abuse monitoring and rate limits.

Quiz · Question 1 of 2
Your pipeline parses model output as JSON and fails ~5% of the time on malformed responses. What's the most robust fix on the OpenAI platform?
A
Add "please return valid JSON" three times to the system prompt and retry on failure
B
Use Structured Outputs with a schema so the API guarantees a conforming response
C
Switch to streaming so you can stop the response as soon as JSON looks complete
D
Raise temperature so the model explores more formatting options until one parses
Correct. Structured Outputs binds the response to a JSON Schema and the API guarantees conformance — eliminating the malformed-JSON class of failures rather than papering over it with prompt-pleading and retries. Streaming doesn't validate structure, and raising temperature makes formatting less reliable.
The robust fix is Structured Outputs with a schema (option B) — the API guarantees conformance, removing the failure class entirely. Repeating "return JSON" still relies on the model's goodwill; streaming doesn't validate; higher temperature makes formatting worse.
Quiz · Question 2 of 2
In the Chat Completions format, you want the model to always behave as a "terse options analyst" across a multi-turn conversation. Where does that instruction belong?
A
Prepended to every user message so the model can't forget it mid-conversation
B
In a system-role message at the start of the messages array
C
In the model's first assistant-role message, as a self-introduction
D
It can't be set via the API; you must fine-tune a model with that persona
Correct. Persistent instructions go in the system-role message — the API's equivalent of the system prompt / CLAUDE.md you know from Lesson 12. Stuffing it into every user message wastes tokens and muddies the request; assistant messages are the model's turns; and no fine-tuning is needed for a persona.
Persistent behavior belongs in the system-role message (option B) — the API analog of a system prompt. Repeating it in user messages is wasteful, assistant messages are for the model's replies, and fine-tuning is overkill for a simple persona.
