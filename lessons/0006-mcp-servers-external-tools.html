MCP Servers

Wire external systems into Claude's tool surface. GitHub, APIs, databases — all reachable without leaving the agent loop.

What MCP Is

Model Context Protocol (MCP) is an open standard that lets external systems expose themselves as tools Claude can call. Instead of Claude writing a curl command to hit your API, an MCP server wraps that API and presents it as a first-class tool — with proper typing, authentication, and error handling.

⟷
MCP Server
⟷
GitHub / Jira / DB / API

From Claude's perspective, MCP tools look exactly like built-in tools (Bash, Read, Edit). Claude just calls them. The MCP server handles authentication, rate limiting, and translating Claude's intent into API calls.

Key Distinction from Bash
You could use Bash to call any API. The difference: MCP gives Claude structured tool descriptions with typed parameters and clear error messages — so Claude uses them correctly without guesswork. Bash is general-purpose; MCP is purpose-built.

Configuring an MCP Server

MCP servers are configured in settings.json under the mcpServers key. Two transport types:

~/.claude/settings.json{
  "mcpServers": {

    // stdio — most common. Spawns a local process.
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    },

    // sse — connects to a running HTTP server
    "my-api": {
      "type": "sse",
      "url": "http://localhost:3001/mcp"
    }
  }
}

Once configured, Claude can call tools from that server as if they were built-in. No special syntax — just tool calls.

Applied to a Real Project
A GitHub MCP server would let Claude create PRs automatically — no copy-pasting. A filesystem MCP with path restrictions to output/ would let Claude write results without needing broad Bash access. MCP is how you give Claude capability with guardrails.

When MCP Is Worth the Overhead

MCP servers add setup cost (configuration, authentication, running a process). Worth it when:

  • You need the same external action repeatedly in agent workflows
  • The external system has auth complexity you don't want Claude solving with Bash
  • You want structured tool schemas so Claude uses the API correctly
  • You're building a multi-agent workflow where subagents need external access

Not worth it when you just need to run a one-off shell command — use Bash.

What to Hold Onto

  1. MCP = structured external tools. Anything you can call via an API, you can expose as an MCP tool. Claude calls it like any built-in tool.
  2. Configure in settings.json under mcpServers. stdio for local processes, sse for HTTP servers.
  3. Claude won't automatically prefer MCP tools. Tell it to in CLAUDE.md: "use the X MCP tool for Y operations."
  4. Worth it for repeated, structured, auth-heavy external actions. Not worth it for one-off shell commands.
