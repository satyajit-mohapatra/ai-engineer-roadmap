The Four Strategies

Write · Select · Compress · Isolate — applied in depth. Every context engineering decision maps to one of these.

The Framework

We introduced the four strategies in Lesson 9. Now that you understand the window (Lesson 15) and the six harness layers (Lessons 10–14), the strategies become concrete design tools rather than abstractions.

Write
Persist context outside the window so it can survive session boundaries. CLAUDE.md, memory files, task files on disk, checkpoint state before compaction fires. Use when: information needs to outlast a session or survive compaction.
Select
Inject only relevant context at inference time. Memory files loaded on demand. Skills only active when triggered. RAG retrieval. Tool calls that return targeted results. Use when: not all context is needed for every task.
Compress
Reduce token count without losing meaning. Pipe verbose Bash output through jq/grep/head. Write memory summaries instead of raw transcripts. Use WebFetch with a specific extraction prompt. Use when: raw output is verbose but only part is needed.
Isolate
Run sub-tasks in separate agent contexts. Subagents via Agent(). Each gets a fresh, focused window. Results return as a summary — not raw context. Use when: a sub-task would bloat or contaminate the main session.

Decision Tree: Which Strategy Applies?

Something needs to be available in a future session?
Yes →Write (CLAUDE.md if stable, memory file if it changes) No → Is it needed for every task, or only some?
Every task →Write to CLAUDE.md Only some →Select (memory file, skill gating, RAG)
Context is large or verbose?
Yes, and you only need part of it →Compress (grep, jq, summarize, targeted fetches) Yes, and it's a separable sub-task →Isolate (subagent with focused context) No, size is fine → No strategy needed — just load it

Where Strategies Combine

Most real problems require more than one strategy. The classic pattern for long-horizon work:

  1. Write a task plan to disk before starting (survives compaction)
  2. Isolate research and analysis into subagents (clean main context)
  3. Compress subagent results into summaries before returning them
  4. Select only the relevant memory files for each phase
  5. Write each completed phase result to disk as a checkpoint
The Diagnostic Lens
When something goes wrong, ask: which strategy is missing? Claude gives inconsistent answers → Write (rules aren't persisted). Context fills up → Compress or Isolate. Sub-task bleeds into main work → Isolate. Claude loads irrelevant context → Select (too much is always-loaded). Most agent failures trace to a missing or wrong strategy.

Quiz · Question 1 of 3
You want Claude to have access to your full API documentation (5,000 tokens) when working on API-related code — but not when working on unrelated frontend code. Which strategy?
A
Write — add it to CLAUDE.md so it's always available
B
Select — put it in a memory file or subdirectory CLAUDE.md that's only loaded when relevant
C
Compress — summarize it down to 500 tokens and add to CLAUDE.md
D
Isolate — run all options work in subagents
Select is the right call. Put the docs in src/api/CLAUDE.md (subdirectory — only loaded when Claude works in that directory) or in a reference memory file that Claude loads when you ask about options. Always-loading 5,000 tokens for context that's only relevant 50% of the time is a costly Write overuse.
This is a Select problem. The docs are too large to always-load (Write overuse) but too detailed to compress. A subdirectory CLAUDE.md at src/api/CLAUDE.md loads only when Claude is working in that directory — exactly the scoping you need.
Quiz · Question 2 of 3
A subagent returns its full research transcript (30,000 tokens) to your main session. What's wrong, and which strategy fixes it?
A
Nothing — 30,000 tokens is within the window limit
B
Compress is missing — the subagent should return a summary, not its full transcript
C
Isolate is violated — subagents shouldn't return anything to the main session
D
Write is missing — the transcript should be saved to disk instead
Compress is the missing strategy. Isolate is working correctly (the research ran in a separate context), but the result flowing back should be a compressed summary — not the raw transcript. Brief the subagent to return a structured summary: key findings, conclusions, recommendations. That might be 500 tokens instead of 30,000. Write is also valid as a backup — save the full transcript to disk and only load sections when needed.
Isolate is working (the research ran in its own context), but the return value needs Compress. Brief the subagent to return a structured summary — not its full transcript. A 30,000-token return defeats the purpose of isolating the research. Good agent design: isolate the work, compress the result.
Quiz · Question 3 of 3
You diagnose: "Claude keeps using the wrong ESLint config path." Which strategy caused this failure?
A
Compress — the config path was summarized and lost detail
B
Isolate — the config path is in a subagent context that the main session can't see
C
Write missing — the correct path was never persisted to CLAUDE.md or memory
D
Select missing — the right config is available but Claude doesn't know to select it
Write is missing. Claude is guessing the path from context — and guessing wrong. The fix: add the correct path to CLAUDE.md ("ESLint config is at .eslintrc.json in the project root — use this path always"). Or write a feedback memory: "Claude has incorrectly used ./src/.eslintrc — the correct path is .eslintrc.json." Write the correct fact once; stop correcting it every session.
This is Write missing. Claude is guessing because the correct path was never persisted. The fix: add it to CLAUDE.md ("ESLint config: .eslintrc.json in project root") or write a feedback memory. Once written, Claude will use the correct path every session without re-correction.

What to Hold Onto

  1. Write: persist to survive sessions. CLAUDE.md (stable) or memory files (changing).
  2. Select: load only what's relevant. Skills on demand, memory selectively, subdirectory CLAUDE.md by location.
  3. Compress: trim verbose output before it lands in the window. grep/jq/head, targeted fetches, summarized subagent returns.
  4. Isolate: separate contexts for sub-tasks. Subagents for research, for large file analysis, for anything that would bloat the main session.
  5. Diagnostic lens: Claude failure → find the missing strategy. Most failures trace to one of these four.
