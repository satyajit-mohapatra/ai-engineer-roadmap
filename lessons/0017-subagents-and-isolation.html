Subagents & Isolation

How to spawn, brief, and orchestrate subagents. The Isolate strategy made concrete.

What a Subagent Is

A subagent is a child Claude instance spawned by the orchestrator (your main session). It runs with its own fresh context window, executes a focused task, and returns a result. From the orchestrator's perspective it's a single tool call that returns a string.

In Claude Code's harness, subagents are spawned with the Agent tool. The system prompt you've been reading describes this explicitly — skills like /teach and /daily-report use subagents internally.

Why This Matters
Each subagent starts with a clean context. It doesn't inherit the orchestrator's long conversation history, its tool results, or its accumulated context debt. It gets a focused prompt and returns a focused result. The Isolate strategy's core value is keeping the orchestrator's context clean while work gets done in parallel.

When to Spawn a Subagent

Spawn whenDon't spawn when
The task would read many files or produce verbose outputThe task is a single tool call that returns a small result
Multiple independent tasks can run in parallelThe task requires the full orchestrator conversation history
You want a second opinion (independent context = independent judgment)The overhead outweighs the isolation benefit
The task is separable and its result can be summarizedThe task is fast and simple

Briefing a Subagent

A subagent starts cold — no memory of this conversation. Its entire world is the prompt you give it. A bad brief produces a bad result. A good brief has four components:

1
Goal — what to produce
Be specific. "Summarize" is weak. "Return a bulleted list of the top 3 signals and their confidence scores" is actionable.
2
Context — what it needs to know
File paths, repo location, relevant constraints. The subagent can't read your conversation — include what it needs.
3
Constraints — what not to do
Tools it shouldn't use, files it shouldn't touch, scope limits. "Read-only — do not write any files."
4
Output format — how to return the result
Structure the return so the orchestrator can parse it without ambiguity. Tables, JSON, bulleted lists — be explicit.

A Real Subagent Brief

This is the difference between a bad brief and a good one for a data-fetching workflow:

WEAK BRIEFLook at the options data files and tell me what you find.

STRONG BRIEFYou are a read-only research agent. Your task:

**Goal**: Identify the top 5 highest-IV tickers from the most recent
options scan output.

**Context**:
- Project root: /path/to/your-project
- Scan outputs are in output/options_*.csv, sorted by date descending
- IV is in the column "impliedVolatility" (0–1 scale, multiply by 100 for %)

**Constraints**:
- Read files only — do not write, edit, or run any scripts
- Use only the most recent output file (highest date in filename)

**Output format**:
Return a markdown table with columns: Ticker | IV% | Expiry | Strike
Then one sentence: which sector dominates and why.

Do not explain your process — just return the table and sentence.
The Principle
A subagent brief should be so complete that you could hand it to a colleague who doesn't know the project and they'd know exactly what to do. The subagent has no context beyond what you give it. Never delegate understanding — include what it needs.

Parallel vs. Sequential Agents

Subagents can run in parallel (independent tasks) or sequentially (each feeds the next). The pattern matters for both performance and correctness:

PatternWhen to useExample
ParallelTasks are independent — no output of one feeds anotherFetch data for service-A AND run analysis on service-B simultaneously
SequentialEach step needs the previous resultExplore → Plan → Implement → Review
Fan-out / fan-inOne input, many parallel analyses, then synthesisScan 10 tickers in parallel, synthesize results

The Explore / Plan / Code / Review Pattern

For complex implementation tasks, a four-agent pipeline produces better results than a single agent attempting everything:

  1. Explore agent — read-only, maps the codebase, returns a summary of relevant files and patterns
  2. Plan agent — takes the Explore summary, returns a step-by-step implementation plan
  3. Code agent — takes the plan, implements it, returns the diff
  4. Review agent — reads the diff independently, returns findings

Each agent starts clean. The Review agent, in particular, benefits from not having seen the Code agent's reasoning — it gives a genuinely independent assessment.

Quiz · Question 3 of 3
Why does a Review agent produce better findings when it hasn't seen the Code agent's reasoning?
A
The Review agent is a different model with different training
B
Review agents are faster without extra context
C
Independent context = independent judgment. Seeing the implementation reasoning primes the reviewer to rationalize rather than critique.
D
Shorter context means more attention to the diff itself
Isolation prevents anchoring bias. When you see someone else's reasoning first, you tend to evaluate the conclusion through their frame — you're more likely to validate than find flaws. An independent reviewer sees only the artifact (the diff), not the justification, and evaluates it on its own merits. This is why human code review works best when the reviewer hasn't heard the implementation rationale first.
Isolation prevents anchoring. When a reviewer sees the implementation reasoning first, they tend to rationalize rather than critique — they evaluate the code through the implementer's frame. An independent context means an independent judgment: the reviewer sees the diff on its own merits, not filtered through "here's why I did it this way."

What to Hold Onto

  1. Subagents start clean. Zero context from your session. Brief them completely — goal, context, constraints, output format.
  2. Parallel for independent tasks, sequential for dependent ones. Fan-out + synthesis for multi-target analysis.
  3. Return summaries, not transcripts. The orchestrator gets a compressed result. Don't let subagent verbosity bloat the main context.
  4. Independent context = independent judgment. Review agents, second opinions, and validation all work better in isolation.
