Embeddings

How you turn meaning into math. The foundation of semantic search, RAG, and recommendation — the next three lessons all rest on this one.

The core idea
An embedding is a list of numbers (a vector) that represents the meaning of a piece of text. Texts with similar meaning get vectors that point in similar directions. That's the whole trick — and it unlocks searching by meaning instead of by keyword.

From text to vector

An embedding model takes text and outputs a fixed-length vector — often 384, 768, or 1536 numbers:

"call option" → [0.02, -0.41, 0.88, …]
"put option" → [0.05, -0.38, 0.85, …]  (close — related meaning)
"pizza recipe" → [-0.71, 0.22, 0.10, …]  (far away — unrelated)

The model was trained so that semantically similar inputs land near each other in this high-dimensional space. "King − man + woman ≈ queen" is the famous illustration: directions in the space encode relationships.

Measuring similarity

To compare two embeddings you measure the angle between them — cosine similarity (1 = identical direction, 0 = unrelated, −1 = opposite). This single number is the engine of every search and retrieval system you'll build.

python — embeddings + cosine similarityfrom openai import OpenAI
client = OpenAI()
def embed(text):
    r = client.embeddings.create(model="text-embedding-3-small", input=text)
    return r.data[0].embedding   # a 1536-d vector

import numpy as np
def cosine(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

Choosing an embedding model

OptionNotes
OpenAI text-embedding-3-small/largeHosted, strong general-purpose, cheap. Easy default.
Cohere Embed, Gemini Embedding, Jina, VoyageHosted alternatives; some specialize (multilingual, long docs, code).
Sentence-Transformers (open, local)Run on your own machine via Hugging Face — privacy + no per-call cost.
Two rules that bite beginners
  1. Embed queries and documents with the same model. Vectors from different models live in different spaces and aren't comparable.
  2. Dimension count is fixed per model. Your vector DB (Lesson 23) must be configured for that exact dimension. Switching models = re-embedding everything.

What embeddings power

  • Semantic search — find documents by meaning, not exact words ("downside protection" matches "hedging strategy").
  • RAG retrieval — fetch the most relevant chunks to ground an answer (Lesson 24).
  • Recommendation / clustering — group or suggest similar items.
  • Classification / dedup / anomaly detection — distance in embedding space = a feature.

A keyword search for "downside protection" misses a document that says "hedging strategy" despite them meaning the same thing. Why do embeddings fix this?
A
Embeddings expand each query into every possible synonym before matching keywords
B
They map both phrases to nearby vectors by meaning, so similarity is high despite no shared words
C
Embeddings store the full text so the search can re-read documents more carefully
D
They lowercase and stem the words so spelling variants match exactly
Correct. The embedding model places semantically similar text near each other in vector space, so "downside protection" and "hedging strategy" have a high cosine similarity even with zero overlapping words. That's the leap from lexical (keyword) to semantic (meaning) search — no synonym lists, stemming, or full-text re-reading involved.
Embeddings work by meaning, not keywords: both phrases map to nearby vectors, giving high cosine similarity without shared words (option B). They don't expand synonyms, store full text for re-reading, or rely on stemming — those are lexical-search tricks.

Correct. Each embedding model defines its own vector space; a vector from Model A and a vector from Model B aren't on comparable axes, so cosine similarity between them is noise.
