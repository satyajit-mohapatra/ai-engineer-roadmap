The Model Provider Landscape

Beyond OpenAI: Anthropic, Google, and the open-weight players. Plus the abstraction layer that lets you switch between them.

Why this is a short lesson
You learned the OpenAI API shape in Lesson 18 precisely so this lesson can be short: every provider below is a variation on the same theme — messages in, completion out, tool calling, streaming. You're learning differences, not new fundamentals.

OpenAI-compatible APIs & OpenRouter

Two facts make multi-provider life easy:

  • "OpenAI-compatible" endpoints — most providers (and local runtimes like Ollama) expose an endpoint that accepts the OpenAI request format. Change base_url + key + model name; keep your code.
  • OpenRouter — a single API in front of hundreds of models across all providers. One key, one format, switch models with a string. Ideal for the tiered routing from Lesson 16 and for comparison shopping.
python — same code, three providers# OpenAI
OpenAI(base_url="https://api.openai.com/v1")
# Local Llama via Ollama
OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
# Anything, via OpenRouter
OpenAI(base_url="https://openrouter.ai/api/v1", api_key=OR_KEY)
# ...then the SAME .chat.completions.create(model="...") call
Where compatibility breaks down
Provider-specific features — Anthropic's extended thinking, Gemini's enormous context, OpenAI's Responses API tools — aren't in the shared OpenAI shape. For those you use the native SDK. Compatibility covers the common 80%; the differentiated 20% needs the real SDK.
The strategic takeaway
Treat models as swappable behind an interface. Don't hard-code one provider's quirks throughout your app — wrap model calls in a thin adapter so you can re-route by cost, capability, or outage. This is plain software engineering applied to a volatile, fast-moving market.
