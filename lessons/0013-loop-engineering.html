Loop Engineering

The shift from writing prompts to designing the systems that write prompts for you.

The Shift

Every lesson so far has assumed a human in the loop: you type a prompt, Claude responds, you read it, you type the next prompt. That's the unit of interaction this whole curriculum has been built around.

Loop engineering is the name for what happens when you stop being that person. Instead of prompting the agent yourself, you design a system — a loop — that prompts the agent on a schedule, checks its work, and only surfaces results when something needs your judgment.

"I don't prompt Claude anymore. I have loops running that prompt Claude." — Boris Cherny, Anthropic (Claude Code lead)
"Build the loop. But build it like someone who intends to stay the engineer, not just the person who presses go." — Addy Osmani, Loop Engineering

The term was popularized in mid-2026 by Addy Osmani, echoing a line from Peter Steinberger: "you should be designing loops that prompt your agents." The leverage point moves from crafting individual prompts to designing the control system that generates prompts over time.

Where It Started: The Ralph Loop

You met "Ralph Loops" briefly in Lesson 9 as a multi-session continuation pattern. The full story: in July 2025, engineer Geoffrey Huntley documented what he called the "Ralph Wiggum loop" — named after the Simpsons character known for confidently wrong answers.

The pattern is deceptively simple: while true; do claude -p "your prompt"; done — a bash loop that keeps invoking an agent against the same goal, with state persisted to disk between iterations (since each invocation gets a fresh context).

The Ralph Wiggum Failure Mode
Huntley's namesake failure: an agent instructed to emit a "done" token only when the task is complete instead emits it early — on a half-finished job. The loop exits, declares victory, and the work is wrong. This is the central risk of all loop engineering: a loop that checks its own work using the same model that did the work will rubber-stamp itself. Every pattern below exists to prevent this.

The Six Components of a Loop

Osmani breaks a production loop into six parts. If this list feels familiar, it should — every component maps directly onto a harness layer or strategy you already know. Loop engineering doesn't introduce new primitives; it arranges the ones from Lessons 1–12 into a system that runs without you.

1
Automations
Scheduled tasks that discover and triage work independently — the "heartbeat" of the loop. Runs on a cadence (every 5 minutes, daily, hourly) without manual prompting.
↳ Maps to: ScheduleWakeup / CronCreate / /loop
2
Worktrees
Isolated working directories so parallel agents don't overwrite each other's changes. Solves "two agents writing the same file."
↳ Maps to: Isolate strategy (Lesson 8/9)
3
Skills
Reusable instruction sets (SKILL.md) encoding project knowledge, so the loop doesn't "re-derive your whole project from zero every cycle."
↳ Maps to: Layer 4 — Skills (Lesson 3)
4
Connectors
MCP-based integrations to issue trackers, databases, Slack, CI — letting loops "act inside your actual environment," not just read/write local files.
↳ Maps to: Layer 3 — MCP servers (Lesson 6)
5
Sub-agents
Separate agents with different instructions, splitting drafting from verification. Prevents "the model that wrote the code" from grading its own work — the direct fix for the Ralph Wiggum failure mode.
↳ Maps to: Subagents / EPCR pattern (Lesson 9)
6
State / Memory
Persistent markdown files (STATE.md) or external boards tracking progress across runs. "The model forgets, the repo doesn't."
↳ Maps to: Layer 6 — Memory + Write strategy (Lesson 5)
Insight
This is why Lessons 1–12 weren't wasted effort even if you never run an unattended loop. A loop is just these six harness components wired together with a scheduler. If any one of them is weak — vague skills, no isolation, no separate verifier — the loop inherits that weakness and runs it on autopilot, at scale, while you're not watching.

Anatomy of One Loop Iteration

1
Trigger fires
Cron schedule, /loop interval, or webhook from a connector (new PR, failed build)
↓
2
Read state
Loop loads STATE.md / run log — what happened last time, what's still open
↓
3
Maker agent works
In an isolated worktree, using project skills, with MCP connectors for external actions
↓
4
Checker agent verifies
A different agent (or different prompt/context) reviews the maker's output against the goal
↓
5
Write state, decide
Update STATE.md. If done → notify human / open PR. If not → loop again. If risky → human gate.

Six Production Patterns

From cobusgreyling/loop-engineering — a practical reference inspired by Osmani and Cherny. These are starting points, roughly ordered by cost and risk.

Pattern Cadence Cost What it does
Daily Triage Once/day Low (L1, report-only) Scans for new issues, failing tests, stale PRs. Produces a summary report — no changes made.
PR Babysitter Every 5–15 min Low–medium Watches open PRs for CI failures or review comments, responds or fixes within scope.
CI Sweeper On build failure Very high Diagnoses and fixes broken builds. High cost because failures can cascade into long debugging loops.
Dependency Sweeper Weekly Medium Updates dependencies, patch-only by default for safety. Opens PRs for review.
Changelog Drafter Per release Low (L1, draft-only) Generates release notes from merged PRs / commit history. Always human-reviewed before publishing.
Post-Merge Cleanup Off-peak, daily Low Removes merged branches, updates local caches, housekeeping that doesn't touch source.

The Phased Rollout: L1 → L2 → L3

Don't deploy a loop at full autonomy on day one. The recommended progression de-risks by building trust in stages:

L1 — Report Only
The loop runs, investigates, and writes a report. It makes zero changes. You read the report and decide what (if anything) to act on. This is where every new loop starts.
L2 — Assisted Fixes
The loop proposes changes — opens a PR, drafts a fix — but does not merge or push to protected branches. A human reviews and approves before anything lands.
L3 — Unattended
The loop acts autonomously within a tightly scoped allowlist (e.g., patch-only dependency bumps, formatting fixes). Reserved for low-risk, high-confidence, narrow-scope tasks only — after L1/L2 have proven the loop's judgment.
Two Risks That Don't Go Away
  • Comprehension debt — code shipped by a loop that no one on the team has read. It accumulates silently and becomes a liability the first time it needs to change.
  • Cognitive surrender — accepting loop output without judgment because it's usually right. The loop amplifies whatever judgment you bring to it, including the absence of judgment.

Building a Loop With Your Current Tools

You don't need new infrastructure — the components map onto tools already available in this environment:

Component Your tool Notes
Automation / scheduling /loop skill, ScheduleWakeup, CronCreate /loop 15m /check-prs for fixed cadence; omit interval to let the model self-pace via ScheduleWakeup
Worktrees Agent(isolation: "worktree") Each spawned agent gets a clean git worktree, auto-cleaned if no changes made
Skills .claude/skills/*.md Encode the loop's repeatable procedure once, reference by name in the loop prompt
Connectors MCP servers (GitHub, Linear, Slack, etc.) Already configured in this session — plugin_engineering_github, plugin_design_linear, etc.
Sub-agents (maker/checker) Agent tool with different subagent_type e.g., general-purpose agent makes the change, separate code-review skill checks it
State / memory A STATE.md in the project, or the memory system from Lesson 5 Write progress + open questions at the end of every iteration 
Example — minimal L1 daily triage loop# Start it (self-paced, checks back when done):
/loop Check for new failing tests in tests/, write a one-paragraph
summary to STATE.md, do NOT fix anything yet.

# Inside the loop, the agent should:
# 1. Read STATE.md for what was true last run
# 2. Run the test suite (compressed via RTK — Lesson 11)
# 3. Diff against last run's failures
# 4. Append findings to STATE.md
# 5. ScheduleWakeup for the next check (1200-1800s for idle ticks)
Start at L1, always
Your first loop should write a report and do nothing else — even if the fix looks trivial. The point of L1 isn't the report; it's building a track record you can point to before you grant the loop write access. Every "L3 unattended" loop in production started as an L1 that ran for weeks first.

The fix is structural, not about trying harder: a separate sub-agent (the "checker") verifies the maker's work using its own pass over the test suite, independent of whatever the maker agent claims.
