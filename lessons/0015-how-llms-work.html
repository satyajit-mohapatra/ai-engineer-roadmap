How LLMs Work

A precise-enough mental model. Not the math — the moving parts you need to reason about every other lesson.

The one-sentence model
An LLM is a function that, given a sequence of tokens, predicts a probability distribution over the next token — and you generate text by sampling from that distribution, appending, and repeating. Everything else is detail.

1. Tokens — the unit of everything

Models don't see characters or words. They see tokens — sub-word chunks produced by a tokenizer. "Tokenization" is roughly "split text into common fragments." A rule of thumb: ~4 characters ≈ 1 token in English; ~750 words ≈ 1,000 tokens.

Tokenization splits text into chunks.

Tokens are the currency of LLMs. You're billed per token, the context window is measured in tokens, and latency scales with tokens. You already met this in Lesson 7 — here's the link to why: tokens are literally the model's input and output alphabet.

2. The Transformer & attention

Modern LLMs are transformers (Vaswani et al., 2017, "Attention Is All You Need"). The core trick is self-attention: for each token, the model computes how much every other token in the context should influence it. This is how it captures meaning that depends on distant words ("the trade I opened yesterday lost money" — "it" later refers to the trade).

Two consequences you'll use constantly:

  • Attention is the reason context matters. The model literally weighs everything in the window against everything else. Junk in the context dilutes attention — the empirical basis for the whole context-engineering discipline (Lessons 7–8).
  • Attention cost grows ~quadratically with context length. Doubling the prompt more than doubles the compute. This is why long contexts are slow and why compression (Lesson 11) pays off.

3. Training vs Inference — two different worlds

TrainingInference
WhenOnce, before release (months, huge GPU clusters)Every time you call the API
What happensWeights are adjusted to predict next tokens well across trillions of tokensWeights are frozen; model just runs forward to predict next token
StagesPre-training → fine-tuning → RLHF/alignmentPrefill (read prompt) → decode (emit tokens one by one)
You controlNothing (unless you fine-tune — Lesson 28)The prompt, the sampling parameters (Lesson 17)
Why this split matters
Because weights are frozen at inference, the model cannot learn from your conversation. It has no memory between calls. Everything it "knows" about your task must be in the context window right now. This single fact is the root of RAG, memory systems, and context engineering. The model isn't learning — you're re-teaching it every call.

4. The generation loop (autoregression)

how a response is producedcontext = [your prompt tokens]
while not done:
    probs = model(context)        # distribution over next token
    next  = sample(probs, temp)   # pick one (Lesson 17)
    context.append(next)          # feed it back in
    if next == STOP: done = True

This is autoregressive generation: each new token is conditioned on everything before it, including tokens the model just produced. Two implications:

  • Hallucination is structural. The model optimizes for plausible next token, not true. A confident, well-formed falsehood is exactly what the objective rewards if it's statistically likely. Grounding (RAG, Lesson 24) is the fix.
  • Output is sequential, so latency scales with output length. Streaming exists because the first token can be shown before the last is computed.

Quiz · Question 1 of 2
A user complains: "I told the chatbot my name in our last session and today it doesn't remember." What is the precise technical reason — and the correct fix layer?
A
The model forgot during training; retrain it on the user's data to fix this
B
Weights are frozen at inference, so there's no cross-call memory; persist it and re-inject into context
C
The temperature was too high; lower it so the model retains facts between sessions
D
The tokenizer dropped the name; switch to a tokenizer that preserves proper nouns
Correct. At inference the weights are frozen, so nothing from one call carries to the next. "Memory" is an engineering construct: you persist the fact (a file, a DB, the memory system from Lesson 5) and re-inject it into the context window on the next call. Temperature and tokenizers are unrelated to cross-session recall.
The model isn't "forgetting" — at inference, weights are frozen and there is no state between calls. The fix is at the engineering layer: persist the fact and re-inject it into the next call's context (option B). Retraining, temperature, and tokenizers don't create cross-session memory.
Quiz · Question 2 of 2
Why does padding a prompt with large amounts of marginally-relevant text tend to hurt answer quality, in terms of how the model actually works?
A
Extra text is ignored entirely, so it has no effect either way on the output
B
It increases temperature automatically, making the model more random in its replies
C
Self-attention weighs all tokens against each other, so noise dilutes attention on the signal
D
The tokenizer fails on long inputs, corrupting the prompt before the model sees it
Correct. Self-attention computes relevance of every token to every other token. Irrelevant filler still participates in those weightings, diluting the attention the model can place on the parts that actually matter. This is the mechanistic justification for context engineering — curate the window, don't just fill it.
The mechanism is self-attention: every token is weighed against every other token, so filler doesn't get cleanly "ignored" — it competes for attention and dilutes the signal (option C). It doesn't change temperature, isn't silently dropped, and doesn't break the tokenizer.
