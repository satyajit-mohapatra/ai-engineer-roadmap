Lesson 12 Tool-Agnostic Harness Engineering ~30 min

Tool-Agnostic Harness Engineering

The four context strategies are universal. The six harness layers are Claude Code's implementation. Every major AI coding tool has equivalents — here's the map.

Strategies vs. Layers

Lessons 1–10 taught you two things that look the same but aren't:

  • Strategies — Write, Select, Compress, Isolate. These are concepts. They apply to any system that takes a context window as input.
  • Harness layers — CLAUDE.md, hooks, skills, memory, MCP. These are Claude Code's implementation of those strategies.

If you switch tools, the strategies survive intact. The layer names change. This lesson maps what carries over and what you need to relearn.

Layer 1: The Project Knowledge File

Every major AI coding tool has a "always-loaded instructions" file at the project level. This is the Write strategy in action — you persist rules to disk so they don't have to be re-stated each session.

Tool File / Location Format Scope options
Claude Code CLAUDE.md (project root) Markdown Global (~/.claude/), project root, subdirectory
Cursor .cursor/rules/*.mdc Markdown + YAML frontmatter (globs, alwaysApply, description) Always-on, glob-triggered, agent-requested. Legacy .cursorrules still works.
GitHub Copilot .github/copilot-instructions.md (repo-wide)
.github/instructions/*.instructions.md (path-scoped)		 Markdown Organization → repository → personal (highest priority)
Windsurf .windsurfrules (project root)
.devin/rules/ or .windsurf/rules/		 Plain text / Markdown Global (Cascade settings panel) or project-level
Aider CONVENTIONS.md (loaded via --read), configured in .aider.conf.yml Markdown Single file, any name — specify with --conventions-file
Token cost applies everywhere
Every tool loads the project rules file on every request. The same rule from Lesson 4 holds: short beats long, specific beats vague, "always-on" rules eat headroom on every turn. Keep the global rules file tight; move verbose guidance into scoped/conditional rules.
AGENTS.md
The Emerging Cross-Tool Standard
Supported by Cursor, Copilot, Aider, Windsurf, Tabnine and more

AGENTS.md started as an OpenAI initiative (August 2025) and has become the closest thing the ecosystem has to a vendor-neutral, cross-tool configuration standard. Place it at the project root. Every major tool reads it.

What belongs in AGENTS.md: build commands, test commands, code style conventions, architectural decisions, framework-specific constraints — everything an agent needs to work in the codebase without asking.

Relationship to tool-specific files: Use AGENTS.md for the portable core. Use tool-specific files (.cursor/rules/, .github/copilot-instructions.md) for tool-specific overrides or features (like Cursor's glob triggers or Copilot's path-specific instructions).

Where Aider fits: Aider's docs explicitly recommend AGENTS.md as an alternative to CONVENTIONS.md for interoperability. Use aider --read AGENTS.md or add read: AGENTS.md to .aider.conf.yml.

Sources: Augment Code AGENTS.md guide · vibecoding.app cross-tool guide · codersera complete guide

Layer 3: MCP Across Tools

MCP (Model Context Protocol) has become the de-facto standard for extending AI tool surfaces. Support is broad, but the config format varies per tool.

Tool Config file Root key Activation scope
Claude Code ~/.claude/settings.json mcpServers Global or project-level .claude/settings.json
Cursor JSON config file (project or global) mcpServers Project or global
VS Code / GitHub Copilot .vscode/mcp.json servers (not mcpServers) Agent mode only. Not available in inline completion.
Windsurf MCP Marketplace (GUI) or config file varies Global — Cascade panel → MCP Marketplace
Aider No native MCP support (as of 2026) — Use shell wrapper scripts for external tools
VS Code gotcha
Copilot uses "servers" as the root key, not "mcpServers". This is the single most common copy-paste mistake when migrating an MCP config from Claude Code or Cursor. Also: MCP tools only work in Copilot's Agent mode — they don't appear in inline completions or chat.

Layer 5: Hooks — The Claude Code Moat

The Pre/PostToolUse hook system from Lesson 2 is Claude Code-specific. No other major tool has an equivalent as of 2026. This is the biggest architectural gap when switching away from Claude Code.

Capability Claude Code Cursor Copilot Windsurf Aider
Block a tool call before it runs ✓ PreToolUse (exit 1) ✗ ✗ ✗ ✗
Run automation after a tool call ✓ PostToolUse ✗ ✗ ✗ ~ shell wrappers
React to session lifecycle events ✓ SessionStart, Stop, etc. ✗ ✗ ✗ ✗
CI/pre-commit wrappers (rough substitute) ✓ (in addition to hooks) ~ ~ ~ ~

Workarounds for other tools: The patterns hooks implement (safety gates, auto-linting, logging) can be partially replicated using pre-commit hooks, CI pipelines, and shell scripts that wrap the tool's CLI. But they're not equivalent — they can't inspect or block individual AI tool calls in real-time.

Skills and Memory — Concept Equivalents

Claude Code concept What it does Nearest cross-tool equivalent
Skills (Layer 4) — .claude/skills/*.md Reusable, slash-command-triggered workflows that inject a full task brief into context Cursor: .cursor/rules/ with alwaysApply: false (agent-requested) · Copilot: no direct equivalent · Aider: aliases in .aider.conf.yml
Memory (Layer 6) — .claude/memory/*.md Persistent cross-session facts injected into future sessions via MEMORY.md index Windsurf: Cascade Memories (automatic, workspace-scoped) · Cursor: rules that accumulate knowledge · Others: manually maintained AGENTS.md sections
Insight
The underlying pattern for memory is the same everywhere: write durable facts to disk, inject them selectively at the start of future sessions. Only Claude Code automates this end-to-end. In other tools, you do the same thing manually — keep a CONTEXT.md or add a section to your AGENTS.md that you update as the project evolves.

Compression Tools: What Travels, What Doesn't

Tool-agnostic — works everywhere
Repomix Packs a repo into a single file you inject into any tool. npx repomix
LLMLingua Pre-processes retrieved docs before you paste them into any tool. Python library.
Graphify Produces GRAPH_REPORT.md you include in any tool's context. npx graphify query
LSP Compiler-backed symbol navigation. Supported in VS Code, Cursor, Zed, any IDE with LSP.
Claude Code-specific — needs a substitute
RTK Hooks into Claude Code's Bash tool via PreToolUse. No hook system = no RTK. Substitute: manually pass compressed output, or use Repomix for file context.
Caveman mode A Claude Code skill (/caveman). Substitute: add an output style rule to your project rules file: "Be terse. Bullet points only. No prose."

The Portable Harness: A Tool-Agnostic Template

This five-layer structure implements all four strategies without depending on Claude Code primitives. Drop it into any tool.

Portable harness template — works with Claude Code, Cursor, Copilot, Windsurf, Aider
1
Write layer — AGENTS.md at project root ↳ Write strategy
Project conventions, build commands, test commands, architectural constraints. Keep it under 400 words. Every tool reads it. This is your portable CLAUDE.md.
2
Write layer — CONTEXT.md (maintained manually) ↳ Write + Select strategy
Cross-session facts: decisions made, constraints discovered, things the tool got wrong. Update it yourself at the end of each session. Include it explicitly when it's relevant. This is your portable memory system.
3
Compress layer — Graphify + Repomix ↳ Compress strategy
Run npx graphify index once. Add a query instruction to AGENTS.md. For whole-codebase tasks, pack with Repomix and hand to a subagent or separate chat session. Replaces RTK for the file/architecture sources of token bloat.
4
Isolate layer — separate sessions per concern ↳ Isolate strategy
Don't mix an architecture discussion with an implementation sprint in the same session. Start fresh chat sessions for different concerns. Pass only the relevant subset of context (a Repomix pack, a Graphify subgraph, a CONTEXT.md excerpt) — never the whole history.
5
Ratchet layer — append to AGENTS.md on every mistake ↳ Ratchet principle
Every time the tool does something wrong, add a rule. "Never use default exports." "Always check for null before accessing .data." "Tests go in tests/unit/, not next to the source." The harness only tightens.

Choosing Your Primary Tool

If you need… Reach for
Hook-based safety gates, blocking bad tool calls Claude Code — only tool with PreToolUse/PostToolUse
Per-file-type rules (different rules for React vs. API files) Cursor — glob-triggered .mdc rules are uniquely powerful
Org-level governance, team shared instructions GitHub Copilot — organization-level instructions propagate to all repos
Automatic session memory without manual upkeep Windsurf — Cascade Memories accumulate context automatically
Model-agnostic, works with any LLM API Aider — plugs into any OpenAI-compatible endpoint
Portability — must work across multiple tools / team with mixed preferences AGENTS.md-first — write your harness in AGENTS.md, use tool-specific files only for overrides

You're moving a project from Claude Code to Cursor. You had a PreToolUse hook that blocked any git push to main without confirmation. How do you replicate this in Cursor?
A
Add a .cursor/rules/git.mdc file with alwaysApply: true that says "never push to main"
B
Configure an MCP server that intercepts git commands
C
You can't replicate it exactly — Cursor has no hook system. Closest substitute: a git pre-push hook (shell-level) plus a rule telling the AI never to push to main directly
D
Use Cursor's MCP Marketplace to find a git safety extension
Correct. The PreToolUse hook is Claude Code-specific and has no direct equivalent in Cursor, Copilot, Windsurf, or Aider as of 2026. The closest you can do is: a shell-level git pre-push hook (which fires before any push at the OS level, regardless of what tool invoked git) plus a rule in your project rules file that instructs the AI never to run that command. It's weaker than a real-time hook — it can't block the AI mid-turn — but it's the best available substitute.
Cursor has no hook system. The PreToolUse/PostToolUse architecture is unique to Claude Code. A rules file instruction ("never push to main") guides the model but doesn't technically block the tool call. The practical answer is: a shell-level git pre-push hook combined with a project rule. Accept that this harness layer is weaker outside Claude Code.

Your team has engineers using Claude Code, Cursor, and GitHub Copilot. You want one set of project instructions that all three tools read. What's the right approach?
A
Maintain three separate files: CLAUDE.md, .cursorrules, and .github/copilot-instructions.md — one per tool
B
Write the shared conventions in AGENTS.md (cross-tool standard), then add thin tool-specific override files for features unique to each tool
C
Put everything in .github/copilot-instructions.md since GitHub is the source of truth for the repo anyway
D
Use MCP to serve the instructions — all tools support MCP so they'll all pick it up
AGENTS.md is the right spine. It's the vendor-neutral standard that Cursor, Copilot, Aider, Windsurf, and Claude Code all read. Write the shared conventions there — build commands, test commands, architecture, the ratchet rules. Then add tool-specific files only where you need tool-specific features: .cursor/rules/ for Cursor's glob triggers, .github/copilot-instructions.md for Copilot's org-level propagation, CLAUDE.md for Claude Code's subdirectory scoping or hooks-related context. The portable core stays in one place.
AGENTS.md is the right answer. It's the cross-tool standard that all three tools (Claude Code, Cursor, Copilot) read. Maintaining three separate files means three places to update when a rule changes — and they drift. The right structure: shared conventions in AGENTS.md, tool-specific features in tool-specific override files. MCP doesn't carry project instructions; it extends tool surfaces with external capabilities.
Quiz · Question 3 of 3
You switch from Claude Code to Windsurf. You had RTK installed for shell command compression, saving ~80% per Bash call. What's the right substitute strategy in Windsurf?
A
Install RTK in Windsurf — it works with any tool via MCP
B
RTK installs as a Windsurf extension from the MCP Marketplace
C
RTK won't work — it relies on Claude Code's hook system. Substitute: use Repomix to pack file context, run LLMLingua on long retrieved docs, and add terse-output rules to .windsurfrules
D
Switch back to Claude Code for any project that needs token compression
RTK requires Claude Code's PreToolUse hook to intercept Bash output before it lands in context. No hook system means no RTK. The best substitute stack in Windsurf: Graphify (query the graph instead of reading files — the biggest win), Repomix (for whole-codebase tasks), LLMLingua (for long retrieved docs), and an explicit rule in .windsurfrules asking for terse output (the manual version of Caveman mode). You lose the automatic shell-output interception, but you can eliminate most of the underlying causes of token bloat through the other tools.
RTK is Claude Code-specific. It intercepts Bash output using Claude Code's PreToolUse hook — without that hook, RTK has nothing to intercept. The portable substitutes are: Graphify (avoid reading files at all), Repomix (controlled file packing), LLMLingua (doc compression), and terse-output rules in your project rules file. You lose automatic shell compression, but the other tools address the root causes.
