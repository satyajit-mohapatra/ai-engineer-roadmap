Open-Source & Self-Hosted Models

Running models you control — on your laptop or your own servers. The privacy axis from Lesson 3, made real.

When you reach for this
Data can't leave your machine · you want zero per-token cost at scale · you need to fine-tune or modify the model · you want to work offline. If none of those apply, a hosted API (Lesson 6) is usually less work.

Hugging Face — the GitHub of models

Hugging Face is where open models, datasets, and demos live. The pieces you'll touch:

ThingWhat it is
The HubHosting for model weights, datasets, and Spaces (demos). Search, filter, download.
ModelsLlama, Mistral, Qwen, Gemma, DeepSeek, embedding models, and 100k+ others — each with a model card.
TasksThe Hub organizes models by task: text-generation, embeddings, ASR, image-classification, etc.
Inference SDKThe transformers library + Inference API/Endpoints to run models locally or hosted.
Transformers.jsRun small models in the browser via WebGPU — no server at all.

Ollama — local models, one command

The fastest way to run an open model on your own machine. It handles download, quantization, and serving with an OpenAI-compatible API.

bash + python# pull and run a model locally
ollama pull llama3.2
ollama run llama3.2 "Summarize theta decay in one line"

# it exposes an OpenAI-compatible endpoint on :11434
from openai import OpenAI
client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")
# ...same .chat.completions.create() call as Lesson 6
The payoff of Lesson 6
Because Ollama (and LM Studio, and many others) speak the OpenAI API shape, the code you learned in Lesson 6 runs against a local model by changing only base_url. This is why "learn the OpenAI shape once" pays off — it's the universal adapter.

LM Studio & self-hosting at scale

  • LM Studio — a desktop GUI for discovering, downloading, and chatting with local models; also serves an OpenAI-compatible endpoint. Great for non-terminal exploration.
  • Production self-hosting — for real throughput you use a serving engine like vLLM or TGI on a GPU box, which batch requests efficiently. This is where "free weights" meets "you pay for GPUs + ops" from Lesson 3.
  • Quantization — shrinking model weights (e.g. to 4-bit) so they fit on consumer hardware, trading a little quality for big memory/speed savings. It's why a 7B–8B model runs on a laptop at all.

Quiz · Question 2 of 2
A 70B-parameter model won't fit in your laptop's memory at full precision. What technique most directly lets you run a smaller-footprint version locally, and what's the trade-off?
A
Streaming — it sends the model in chunks so it never needs full memory at once
B
Quantization — store weights at lower precision; trade a little quality for memory/speed
C
Few-shot prompting — adding examples reduces the model's memory requirements
D
Raising temperature — a hotter model compresses its own weights automatically
Correct. Quantization reduces the numeric precision of the weights (e.g. 16-bit → 4-bit), shrinking the memory footprint enough to fit on consumer hardware, at the cost of a small quality drop. Streaming is about output delivery, and prompting/temperature don't change a model's size.
Quantization (option B) lowers weight precision to shrink the footprint, trading a bit of quality for fitting in memory. Streaming concerns output, not model size; few-shot and temperature don't reduce memory at all.
