Prompt Engineering Techniques

The named patterns: zero-shot, few-shot, chain-of-thought, ReAct. You know context engineering — these are the discrete tools inside it.

Prompt vs context engineering
Prompt engineering = crafting the wording of a single instruction. Context engineering (your Phase 1 depth) = managing everything in the window over time. Prompt techniques are the building blocks; context engineering is the architecture. This lesson sharpens the blocks.

Zero-shot vs Few-shot

  • Zero-shot — just ask. No examples. Works when the task is common and the instruction is clear. ("Classify this review as positive or negative.")
  • Few-shot — include a handful of input→output examples in the prompt. The model infers the pattern. The single most reliable way to control format and edge-case handling without code.

Why few-shot works
The examples occupy the context window and self-attention (Lesson 2) lets the model pattern-match the new input against them. You're not training the model — you're demonstrating the task in-context. 2–5 well-chosen, diverse examples usually beat 20 redundant ones.

Chain-of-Thought (CoT)

For multi-step reasoning, ask the model to show its work before answering. The classic trigger: "Let's think step by step." Reasoning in the output gives later tokens more relevant context to condition on (Wei et al., 2022).

CoTQ: A spread costs $2.40 to open and the max value is $5.
   What's the risk/reward? Think step by step, then state the ratio.
CoT in 2026
Modern reasoning models (o-series, Claude extended thinking, DeepSeek-R1) do CoT internally — you often don't need to prompt for it, and forcing it can even hurt. Reserve explicit CoT for standard (non-reasoning) models, or when you want the reasoning visible for debugging/auditing.

ReAct — Reason + Act

ReAct (Yao et al., 2022) interleaves reasoning steps with actions (tool calls) and observations (tool results): Thought → Action → Observation → Thought → …. It's the prompting pattern underneath most agents — the model reasons about what to do, calls a tool, reads the result, and reasons again. You'll build on this directly in Lesson 26.

Structuring the prompt itself

LeverEffect
Role ("You are a risk-averse options analyst")Sets tone, vocabulary, and priorities
System promptingPersistent rules that outrank the user turn (Lesson 12 / 6)
Delimiters (XML tags, ###, triple quotes)Cleanly separate instructions from data — reduces injection & confusion
Output constraints ("Respond in ≤3 bullets, no preamble")Controls shape and length
Positive instructions ("Do X") over negative ("Don't do Y")Models follow "do" more reliably than "don't"
Robust prompting = make it hard to misread
Put instructions and untrusted data in clearly different sections (e.g. wrap user content in <data>…</data>). This is both a quality and a security move — it's your first line of defense against prompt injection (next lesson).

You need the model to output trade signals in a very specific custom format it keeps getting slightly wrong with a plain instruction. What's the most reliable low-code fix?
A
Repeat the format rule five times in the system prompt until it sticks
B
Add a few-shot block of 3 input→output examples in the exact target format
C
Raise the temperature so the model varies until it lands on the right format
D
Fine-tune the model on the format before using it for anything else
Correct. Few-shot examples are the highest-leverage, lowest-effort way to lock a format — the model pattern-matches the demonstrated shape. Repetition is weaker, higher temperature makes format less stable, and fine-tuning is heavy machinery for what 3 examples solve. (For machine-readable formats, also consider Structured Outputs from Lesson 6.)
Few-shot examples (option B) are the reliable, low-code fix — demonstrate the exact format 3 times and the model matches it. Repetition is weak, higher temperature destabilizes format, and fine-tuning is overkill here.
