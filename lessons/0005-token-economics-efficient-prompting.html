Token Economics & Efficient Prompting

Tokens are the unit you're billed in and the unit that determines latency. Before you reach for compression tools, these are the cheap, deliberate habits that keep the bill — and the wait — small.

Recall from Lesson 2
Generation is sequential: the model emits one token, feeds it back in, emits the next. That's why output tokens cost more and take longer than input tokens — input is processed in parallel in a single forward pass; output can't be. Every lever below follows from that one fact.

The two budgets: cost and latency

Every API call has two token counts that matter separately:

  • Input tokens — everything you send: system prompt, conversation history, retrieved documents, tool results. Priced lower per-token, processed in parallel, so it mostly affects cost, not speed.
  • Output tokens — everything the model generates. Priced higher per-token (often 3–5×), and generated one at a time, so it drives both cost and the time you wait.

This means the highest-leverage move is usually shrinking output, not input — a lesson most people learn backwards.

Prompt caching — pay once, reuse many times

If the same large block of text (a system prompt, a style guide, a set of docs) appears at the start of every call in a session, both Anthropic and OpenAI let you cache it. A cache hit reprocesses none of that text — you pay a small fraction of the input price and the response starts faster.

The one rule that makes caching work
Put everything static first (system prompt, reference docs, tool definitions) and everything variable (the user's actual question) last. If a single token changes upstream of the cached block, the whole block misses. Stable ordering is the entire trick. 
Anthropic — explicit cache breakpoint{
  "system": [
    {"type": "text", "text": long_static_docs,
     "cache_control": {"type": "ephemeral"}}
  ],
  "messages": [{"role": "user", "content": question}]
}
// First call: pays full price, writes the cache.
// Calls within the TTL window: ~90% cheaper on the cached portion.

The cache has a short TTL (minutes, not hours) — it rewards tight loops of related calls, not occasional ones. If your calls are spaced far apart, caching won't help; restructure for batching instead (below).

Send less input on purpose

This is different from Lesson 19's compression tools, which automatically shrink tool output after the fact. This is about not generating the bloat in the first place:

  • Excerpt, don't paste — if you need lines 40–60 of a file, send those lines with a line-number anchor, not the whole file.
  • Summarize old turns — in a long conversation, the first ten exchanges are rarely load-bearing for the eleventh. Periodically replace history with a short recap.
  • Don't echo what the model just wrote — if Claude just produced a 500-line file, you don't need to paste it back to ask a follow-up question about it; it's already in context.
  • Reference, don't re-send, static material — if a doc is already cached or already in context, point at it ("the schema above") instead of repeating it.

Bound the output deliberately

Recall Lesson 4's max_tokens and stop sequences — they're cost controls as much as formatting controls, since output is the expensive half of every call.

  • Ask explicitly for the shape you want: "answer in 3 bullet points," "return only the JSON, no explanation." Verbosity is a default, not a requirement.
  • Set max_tokens tight for short, structured answers; generous only when you actually expect a long answer.
  • Use a stop sequence to cut off generation right after the structured part ends, before the model adds an unrequested summary.

Match the model to the task

Not every step in a pipeline needs your most expensive model. A classification, extraction, or routing step that a small/cheap model handles at 95% accuracy doesn't need a frontier model — save that budget for the one step that actually requires deep reasoning. Lesson 3 covers the decision framework; the token-economics version of it is simply: the model is also a cost lever, not just a quality lever.

Check Your Understanding

Quiz · Question 1 of 2
A request and its response both run about 500 tokens, but the response consistently takes longer to generate and costs more per token. Why, fundamentally?
A
Output tokens are larger in byte size than input tokens
B
Input is processed in parallel in one pass; output is generated one token at a time, feeding back in
C
Output always passes through a slower, separate model
D
Pricing is arbitrary and unrelated to compute cost
Correct. Input tokens are processed together in a single forward pass — that's parallelizable and cheap. Output tokens are generated sequentially: each one requires its own forward pass before the next can start. That sequential dependency is why output is both slower and priced higher per token — it's a direct compute cost, not an arbitrary pricing choice.
It comes down to how generation works (option B): input is processed in parallel in one pass, but output tokens are produced one at a time, each depending on the last. That sequential step is real extra compute per token — which is exactly why output is priced higher and takes longer, not because of token size or a separate model.
Quiz · Question 2 of 2
You're sending the same 4,000-token style guide as part of the system prompt on every call in a rapid back-and-forth session. What's the most direct fix, and what's the one rule that makes it work?
A
Switch to a cheaper model; the style guide cost is unavoidable otherwise
B
Prompt caching; the static block must come first and stay byte-identical across calls
C
Lower max_tokens on the response so there's room left for the style guide
D
Run LLMLingua on the style guide before every call
Correct. This is exactly what prompt caching is for: a large, unchanging block sent repeatedly in a tight loop. Put it first in the prompt, keep it byte-identical call to call, and cache hits cost a fraction of the full input price. The rule that makes it work is ordering — static content first, variable content (the actual question) last, since any change upstream of the cached block invalidates the cache.
Prompt caching (option B) is the direct fix — a large static block repeated across a tight sequence of calls is the textbook case. The rule: static content first, unchanged byte-for-byte, variable content after it. Changing the model doesn't address the redundant resend; max_tokens only bounds output, not the cached input; LLMLingua is for compressing retrieved/variable text, not for avoiding redundant resends of a fixed block.
